# Session Notes -- 2026-05-19

## Overview

Debugging session for Gemma 4 26B MoE QLoRA training on a single RTX PRO 6000 WS (96 GB VRAM).
Outcome: switched active training to Gemma 3 27B; built a MoE pre-quantization pipeline for future use.

---

## Issues Fixed

### 1. PEFT LoRA injection failure -- `Gemma4ClippableLinear`

**Error:**
```
ValueError: Target module Gemma4ClippableLinear (...) is not supported.
```

**Cause:** Gemma 4 wraps attention projection layers (`q_proj`, `k_proj`, `v_proj`, `o_proj`) in a
custom `Gemma4ClippableLinear` class that contains a `Linear4bit` as a child `.linear`. PEFT only
recognises standard `nn.Linear` / `Linear4bit` as injection targets.

**Fix:** Changed `target_modules` in all qlora config sections to target the inner child:
```yaml
- "q_proj.linear"   # was "q_proj"
- "k_proj.linear"
- "v_proj.linear"
- "o_proj.linear"
```

PEFT matches by suffix of the module path in `named_modules()`, so `"q_proj.linear"` resolves to the
inner `Linear4bit` which is supported.

### 2. CUDA OOM during training forward pass

**Error:**
```
torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 1.51 GiB.
GPU 0 has a total capacity of 94.97 GiB of which 1.45 GiB is free.
```

**Cause (primary):** `WeightedSFTTrainer.compute_loss` called `model(..., labels=inputs["labels"])`.
Transformers' internal loss function immediately casts the full logits tensor to float32:
```python
logits = logits.float()  # transformers/loss/loss_utils.py
```
For shape `[2, 4096, 256K vocab]` this is a ~8 GB spike -- entirely wasted since we compute our own
weighted loss immediately after.

**Fix:** Removed `labels` from the model call. Both weighted and unweighted paths now compute loss via
`CrossEntropyLoss` in bf16, staying within the existing memory budget.

```python
# before
outputs = model(input_ids=..., attention_mask=..., labels=inputs["labels"])
loss = outputs.loss  # triggers logits.float() internally

# after
outputs = model(input_ids=..., attention_mask=...)  # no labels
# compute loss ourselves in bf16
```

**Fix (secondary):** Reduced `max_length` from 4096 to 3072 in the `cloud` profile to lower
activation/logits baseline memory.

---

### 3. Model weights consuming ~95 GB -- no headroom for training

**Finding:** Confirmed 426 `Linear4bit` layers (bitsandbytes 4-bit IS working for attention).
The problem is architectural: MoE expert FFN weights are stored as packed 3D `nn.Parameter` tensors:
```python
self.gate_up_proj = nn.Parameter(torch.empty(num_experts, 2 * intermediate_dim, hidden_dim))
self.down_proj    = nn.Parameter(torch.empty(num_experts, hidden_dim, intermediate_dim))
```
bitsandbytes only quantizes `nn.Linear` submodules -- these Parameters are invisible to it and stay
in bf16, consuming the bulk of VRAM.

**Also found:** `Gemma4ForConditionalGeneration` loads a vision tower + audio tower that are unused
for text-only SFT. Added `_offload_unused_towers()` in `model_loader.py` to move them to CPU after
load and free GPU memory.

**Decision:** Single 96 GB GPU cannot fit this MoE model for QLoRA training. Switched to
`google/gemma-3-27b-it` (dense, fully quantizable) for active training work.

## MoE Expert Quantization -- Future Work

Built a complete pipeline to make the Gemma 4 MoE trainable on a single 96 GB GPU.

### Root problem
The `Gemma4TextExperts` class stores all expert weights as two 3D Parameters.
bitsandbytes cannot quantize them. For 128 experts across 30 layers, this is ~40-50 GB in bf16.

### Solution: monkey-patch + pre-quantization cache

**`moe_quant_patch.py`**
- `_QuantizedExperts`: drop-in replacement for `Gemma4TextExperts` that stores expert weights as
  `nn.ModuleList` of per-expert `Linear4bit` modules instead of 3D Parameters.
  Forward pass is functionally identical -- slices and calls each expert's `Linear4bit` layer.
- `patch_moe_experts_to_4bit(model)`: walks the model, finds all `Gemma4TextExperts` blocks,
  builds the quantized replacement. Deletes original Parameters before swapping to minimise peak
  memory. Returns count of patched blocks.
- `load_expert_cache(model, cache_dir)`: loads pre-built NF4 expert weights from disk instead of
  re-quantizing at startup.

**`convert_gemma4_moe.py`** + **`convert_gemma4_moe.sh`**
- Offline conversion: load model with bitsandbytes 4-bit, apply monkey-patch, save quantized expert
  weights to a sidecar cache (~11 GB).
- Cache format: `expert_cache.pt` (torch.save, CPU tensors) + `manifest.json`.
  Each entry keyed by module path, contains per-expert `uint8` weight tensors and `QuantState` dicts.
- Includes `--verify` flag to check an existing cache without re-converting.

**`model_loader.py` integration**
- `_patch_moe_experts()` called after load, before `prepare_model_for_kbit_training`.
- If `model.moe_cache_dir` is set in config, loads from cache; otherwise quantizes fresh.
- No-op on non-MoE models (Gemma 3 27B) -- safe to leave in the loading path.

### To run conversion (when ready):
```bash
./convert_gemma4_moe.sh --output-dir ./gemma4_moe_quant_cache
./convert_gemma4_moe.sh --verify --output-dir ./gemma4_moe_quant_cache
```

### To train MoE with cache:
```yaml
# configs/default.yaml
model:
  name: "google/gemma-4-26b-A4B-it"
  moe_cache_dir: "./gemma4_moe_quant_cache"
```

### Memory profile (after patch):
| Stage | GPU memory |
|---|---|
| Model load (attention 4-bit, experts bf16) | ~47-50 GB |
| During expert patching (peak per block) | ~52 GB |
| After patch (all layers 4-bit) | ~14-16 GB |
| Training headroom | ~80 GB |

### Caveats
- Conversion requires GPU (bitsandbytes CUDA kernels).
- `@use_experts_implementation` decorator on `Gemma4TextExperts` may dispatch to a CUDA kernel
  implementation for speed. `_QuantizedExperts` uses the pure Python loop -- correct but may be
  slower. CUDA kernel path would need separate adaptation if throughput is critical.
- The AWQ pre-quantized checkpoint (`cyankiwi/gemma-4-26B-A4B-it-AWQ-4bit`) was tested and found
  incompatible: expert weight keys mismatch current transformers architecture (MISSING in load
  report). Not usable for training.
